### Q4：系统中JWT认证是如何实现的？Token过期了怎么办？
**参考答案：**
- 使用Hutool的JWT工具类生成和解析Token
- Token包含userId、role、expireAt三个核心字段
- 有效期设置为72小时，在`application.yml`中可配置
- 前端请求时通过`Authorization: Bearer <token>`头部传递
- 后端`AuthInterceptor`拦截器统一验证Token有效性
- **Token过期处理**：前端检测到401状态码后，自动跳转登录页，用户重新登录获取新Token
- 当前未实现Token自动刷新机制，这是可以优化的点

**代码位置**：

### Q5：系统中是如何处理跨域问题的？
**参考答案：**
- 通过Spring Boot的`CorsConfig`配置全局跨域
- 允许所有来源（`allowedOriginPatterns("*")`）
- 支持GET/POST/PUT/DELETE/OPTIONS方法
- 允许携带凭证（`allowCredentials(true)`）
- 预检请求缓存1小时

```java
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
                .allowedOriginPatterns("*")
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
                .allowedHeaders("*")
                .allowCredentials(true)
                .maxAge(3600);
    }
}
```

## 三、数据库设计类问题

### Q7：数据库中为什么使用逻辑删除而不是物理删除？
**参考答案：**
- **数据安全**：误删后可恢复，避免数据永久丢失
- **审计需求**：保留操作痕迹，便于后续追溯
- **关联数据保护**：删除用户时，保留其订单、帖子等历史记录
- **实现方式**：每个表都有`deleted`字段（0=未删除，1=已删除），MyBatis Plus自动过滤已删除数据

---

### Q8：用户密码是如何存储的？安全吗？
**参考答案：**
- 当前使用**MD5加密**存储密码
- 注册时对明文密码进行MD5哈希后存入数据库
- 登录时对输入密码同样MD5哈希后与数据库比对
- **存在的问题**：MD5已被证明不够安全，容易被彩虹表攻击
- **改进方案**：应使用BCrypt等慢哈希算法，增加盐值（Salt）

```java
// 当前实现（MD5）
user.setPassword(DigestUtil.md5Hex(userRegisterDTO.getPassword()));

// 建议改进（BCrypt）
// user.setPassword(BCrypt.hashpw(password, BCrypt.gensalt()));
```

### Q9：数据库表命名为什么有的用`yl_`前缀，有的不用？
**参考答案：**
- 项目初期部分表没有统一前缀（如`sys_user`、`address`、`favorite`）
- 后期为了区分业务表和系统表，新增了`yl_`前缀（如`yl_material`、`yl_order`、`yl_post`）
- 这是项目迭代过程中的历史遗留问题
- **改进建议**：统一命名规范，所有业务表添加相同前缀

## 四、AI功能类问题

### Q10：DeepSeek AI是如何集成的？调用流程是怎样的？
**参考答案：**
- 通过HTTP请求调用DeepSeek OpenAI兼容API
- 使用Hutool的`HttpRequest`发送POST请求
- 请求体包含：model、messages（含system prompt和user message）、temperature、max_tokens
- 解析返回的JSON，提取AI回复内容

**调用流程**：
1. 用户输入问题 → 2. 构建Prompt（System Prompt + 历史对话 + 当前问题） → 3. 发送HTTP请求到DeepSeek API → 4. 解析响应 → 5. 保存对话记录 → 6. 返回给用户

### Q11：System Prompt的作用是什么？如何防止AI乱回答？
**参考答案：**
- **System Prompt**用于设定AI的角色和行为规则
- 本项目中设定了以下核心规则：
  1. 只回答考研相关问题
  2. 不编造平台客服联系方式
  3. 不推荐具体资料名称和价格
  4. 不编造平台没有的功能
- **防护措施**：
  - 严格的System Prompt约束
  - 后端对AI回复进行关键词过滤
  - 敏感操作（如退款、投诉）统一引导到官方渠道

### Q12：AI推荐功能是如何实现的？推荐算法是什么？
**参考答案：**
- **混合推荐策略**：

  2. **协同过滤**：基于用户行为数据（浏览、收藏、购买）找相似用户
  3. **AI智能推荐**：调用DeepSeek分析用户画像，生成个性化推荐理由
- **实现流程**：
  1. 收集用户行为数据（UserBehavior表）
  2. 构建用户画像（目标院校、专业、学习偏好）
  3. 获取候选资料列表
  4. 调用AI分析并生成推荐理由
  5. 返回带推荐理由的个性化列表

### Q13：如果DeepSeek API调用失败或网络异常，系统如何处理？
**参考答案：**
- 配置了`ai.deepseek.enabled`开关，可一键禁用AI功能
- API调用使用try-catch捕获异常
- 失败时降级为**默认推荐策略**（热门推荐、最新上架）
- 返回友好的提示信息，不暴露技术细节

```java
if (aiEnabled && StrUtil.isNotBlank(apiKey)) {
    try {
        String aiResult = callAiForRecommend(...);
        // 解析AI结果
    } catch (Exception e) {
        // 降级处理：使用默认推荐
        result.setRecommendReason("为你推荐热门资料");
    }
} else {
    // AI未启用，使用默认推荐
    candidates = getPopularMaterials(limit);
}
```

### Q15：系统中如何实现权限控制的？不同角色能看到什么？
**参考答案：**
- **三级角色体系**：
  - `STUDENT`：普通用户，使用前台功能
  - `OPERATOR`：运营人员，管理内容和订单
  - `ADMIN`：管理员，拥有全部权限
- **实现方式**：
  1. 自定义注解`@RequireRoles({"ADMIN"})`标注在Controller方法上
  2. `AuthInterceptor`拦截器解析Token中的role字段
  3. 对比注解要求的角色与用户实际角色
  4. 不匹配则抛出"无权限执行该操作"异常
- **前端控制**：根据用户角色动态渲染菜单和按钮

### Q16：文件上传功能是如何实现的？上传的文件存在哪里？
**参考答案：**
- 使用Spring Boot的MultipartFile接收文件
- 文件存储在本地文件系统（`${user.dir}/files/`）
- 通过`WebMvcConfig`将`/files/**`映射到本地目录
- 数据库只存储文件访问URL
- 限制上传文件大小为50MB

**存在的问题**：
- 本地存储不适合生产环境
- **改进方案**：使用OSS（阿里云/腾讯云对象存储）或MinIO

## 九、其他可能问题

### Q25：项目中遇到的最大困难是什么？如何解决的？
**参考答案（示例）**：
- **困难1：AI回答不稳定**
  - 现象：AI有时会编造资料信息或推荐不存在的书籍
  - 解决：设计严格的System Prompt，明确禁止行为；后端增加关键词过滤；对敏感问题统一回复模板
- **困难2：前后端联调跨域问题**
  - 现象：前端请求后端接口报CORS错误
  - 解决：配置全局CorsConfig，允许跨域携带凭证
- **困难3：支付宝回调调试**
  - 现象：本地无法接收支付宝异步通知
  - 解决：使用内网穿透工具（natapp）暴露本地服务
