## 5.1 关键技术实现

### 5.1.1 沙箱支付

为保障系统支付功能的安全性与可测试性，本“研路无忧”考研服务平台采用支付宝沙箱支付技术模拟真实支付环境，无需发生实际资金交易，即可安全、高效地调试并验证所有与支付相关的业务流程，有效规避了测试过程中可能出现的资金风险与操作失误。在实际调试工作开展前，相关人员需先注册支付宝沙箱账号，登录沙箱环境后即可进入虚拟支付平台，该平台内置虚拟买家账号、商家账号及对应支付接口，可完整复现真实支付的全流程，其注册界面如图23所示。

图23 注册沙箱支付

系统通过调用支付宝第三方支付接口，成功实现了网站端支付流程的模拟效果，确保从支付发起、页面跳转至支付结果反馈的全链路运行顺畅，完全符合真实支付场景的预期要求。沙箱支付的时序图如图24所示，该图清晰呈现了用户、系统后端与支付宝沙箱环境之间的交互步骤、数据传递逻辑及各环节的执行顺序，为技术调试与流程优化提供了直观参考。

图24 沙箱支付时序图

支付功能的核心逻辑在 `pay` 方法中实现，该方法主要负责接收并处理前端发起的支付请求，具体参数包括：字符串类型的 `orderNo`（对应订单编号）、字符串类型的 `userId`（对应发起支付的用户ID），以及 `HttpServletResponse` 对象（用于向前端输出支付响应内容）。方法执行流程如下：首先，根据传入的 `orderNo` 调用 `orderService.getByOrderNo(orderNo)` 方法查询对应的订单信息，从中提取订单总金额（`totalAmount`）及订单标题；为避免浮点数运算产生的精度误差，采用 `BigDecimal` 类精确计算支付金额，保障金额计算的准确性，杜绝资金统计偏差。

计算完成后，初始化 `AlipayClient` 实例，用于与支付宝沙箱API进行通信，初始化过程中需配置必要的参数，包括应用ID、商户私钥、数据格式及字符集等核心配置信息，确保通信链路安全且数据传输规范；接着，创建 `AlipayTradePagePayRequest` 请求对象，依次设置异步通知地址（`notifyUrl`，用于接收支付宝支付状态变更的异步通知）、订单号（`out_trade_no`）、订单金额（`total_amount`）、商品标题（`subject`）及产品代码（`product_code`）等请求参数，完成支付请求的构建，具体实现代码如图25所示。

图25 沙箱支付初始化代码

支付请求参数配置完成后，调用 `alipayClient.pageExecute(request)` 方法执行支付请求，生成支付所需的HTML表单内容；随后，设置 `HttpServletResponse` 的响应内容类型为HTML格式，将生成的表单HTML内容写入响应输出流，最后冲刷并关闭输出流，确保前端能够正常接收并渲染支付表单，引导用户完成支付操作，该部分实现代码如图26所示。

图26 沙箱支付生成表单代码

当支付宝沙箱环境中的支付状态发生变更时（如支付成功、支付失败），会向系统预设的异步通知地址（`notifyUrl`）发送POST请求，系统通过该接口处理支付状态回调。具体处理逻辑为：首先从请求参数中提取 `trade_status` 字段，判断其值是否为 "TRADE_SUCCESS"，若为该值则表明用户已在支付宝端成功完成支付；随后，调用 `orderService.updateStatus` 方法，将该订单的状态更新为“已支付”（1），完成交易流程闭环；为保障异步通知的安全性，避免非法请求伪造，可将请求参数封装为 Map 集合，通过 `AlipaySignature.rsa256CheckContent` 方法验证请求签名的合法性，验证通过后再执行后续业务逻辑，确保通知来源真实、数据完整，有效防范恶意攻击，该部分异步回调代码如图27所示。

### 5.1.2 OSS对象存储

考研服务平台需存储大量考研资料封面、资料附件（如PDF、视频）、用户头像及帖子图片，若直接存储在本地服务器，会占用大量服务器资源，导致系统响应速度下降，影响用户使用体验。为解决这一问题，本平台选用阿里云OSS进行文件存储，阿里云OSS是一种高效、安全的云存储服务，专为海量数据存储设计，适用于存储各类非结构化数据，可显著提升存储效率，同时确保用户能够快速访问所需的资料和图片内容，实现流畅的使用体验，图28为注册阿里云账号并创建OSS服务的界面。

图28 创建OSS服务

通过调用阿里云OSS第三方接口，系统实现了用户文件向云服务器的上传功能，有效避免文件存储占用本地服务器过多内存，降低服务器运行压力。用户上传文件到阿里云OSS的时序图如图29所示，清晰展示了整个文件上传过程中的交互步骤、数据传递路径及各参与方的执行逻辑。

图29 上传文件流程图

在 `AliOSSUtils` 工具类中，定义了将图片、资料附件上传到阿里云OSS的核心方法。该工具类定义了四个必要的成员变量：`endpoint` 指定了OSS的访问地址为阿里云杭州区域，`accessKeyId` 和 `accessKeySecret` 用于身份验证，确保上传操作的合法性，`bucketName` 指定了文件要上传的存储桶名称为 "yanluwuyou-resources"，具体配置如图30所示。

图30 配置上传路径

在 `upload` 方法中，接收一个 `MultipartFile` 对象（用于接收前端上传的文件）和一个整数类型的 `type`（用以区分不同的文件类型，便于分类存储）。方法执行流程如下：首先通过 `MultipartFile.getInputStream()` 获取文件的输入流，并通过条件判断构造文件名，文件名中包含当前日期和原始文件名，确保文件名唯一，同时将不同类型的文件分别存储在对应的目录下（如 `avatar/`、`cover/`、`material/`、`post/`），实现文件分类管理；然后通过 `OSSClientBuilder` 创建一个OSS客户端实例，使用提供的 `endpoint`、`accessKeyId` 和 `accessKeySecret` 进行身份认证；随后，调用 `ossClient.putObject(bucketName, fileName, inputStream)` 将文件上传到指定的存储桶中；文件上传成功后，生成一个格式为 `https://bucketName.endpoint/fileName` 的URL，方便用户直接访问所上传的文件；最后，通过 `ossClient.shutdown()` 关闭OSS客户端以释放资源，避免资源浪费，之后返回生成的文件URL，具体实现如图31所示。

图31 配置上传文件夹

当用户调用上传接口上传文件后，文件会自动上传至阿里云OSS中，系统按用户头像、资料封面、资料附件以及帖子图片等类型对文件进行分类存储和展示，便于管理和访问，如图32所示。

### 5.1.3 DeepSeek 智能助理

为提升用户使用体验，增强平台交互性，本平台引入 DeepSeek 作为智能考研助理。DeepSeek 基于先进的人工智能算法，能够为用户提供个性化备考咨询及智能问答服务，当用户与智能助理对话时，DeepSeek 会结合平台内的考研指南、资料信息等数据，精准解答用户的备考疑惑。此外，DeepSeek 持续优化推荐算法，在满足用户基础咨询需求的同时，能够更便捷、高效地挖掘平台内的优质内容，进一步提升用户粘性。

使用 DeepSeek 智能助理前，需注册 DeepSeek 账号并进入 DeepSeek 开放平台，该平台提供了调用 DeepSeek 的详细接口文档，注册成功后，用户可在平台查看自身的消费金额、API请求次数、tokens使用情况等相关信息，如图33所示。

### 5.1.4 协同过滤算法

协同过滤算法是本平台个性化资料推荐功能的核心，其实现过程主要分为三个关键步骤，分别为构建数据模型、解析数据模型及生成推荐列表，具体实现如下：

首先，在 `getDataModel` 方法中，从 `Recommend` 对象的列表中构建包含用户偏好的数据模型，返回类型为 `DataModel`。方法执行流程如下：首先判断传入的推荐列表 `allRecommend` 的大小，若列表为空或为 null，则直接返回 null，避免无效运算；接下来初始化一个 HashMap，该 Map 以用户 ID 作为键，以用户对应的资料偏好（即资料 ID 和推荐分数）作为值，用于存储每个用户的资料收藏及偏好信息；随后遍历 `allRecommend` 列表中的每个推荐项，提取出用户 ID、资料 ID 和推荐分数；对于每个推荐项，若推荐分数不为零，则构建一个 `GenericPreference` 对象并将其添加到对应用户的偏好列表中；若该用户 ID 已存在于 HashMap 中，则将新生成的偏好列表与已有的偏好列表合并；若不存在，则直接将此偏好列表作为新条目加入 Map；完成用户偏好收集后，初始化一个 `FastByIDMap`，用于将最终的用户偏好数据存储在内存中，提升数据访问效率；遍历之前构建的 HashMap，将每个用户的偏好列表转化为 `GenericUserPreferenceArray` 并放置到 `FastByIDMap` 中；最后，使用该偏好映射创建一个 `GenericDataModel` 实例，作为协同过滤算法的基础数据模型并返回，具体实现如图38所示。

图38 构建数据模型代码

然后，在 `baseItem` 方法中，基于构建好的数据模型为指定用户生成推荐资料 ID 列表，该方法输入参数包括用户 ID（`userId`）和代表用户及其偏好的数据模型（`model`）。方法执行流程如下：首先检查 `model` 是否为 null，若为 null 则直接返回 null，确保后续操作不引发空指针异常；接着借助 `UncenteredCosineSimilarity` 类实例化一个物品相似度计算对象 `similarity`，用于计算不同资料之间的相似度；随后基于提供的数据模型 `model` 和相似度对象 `similarity`，创建一个 `GenericItemBasedRecommender` 推荐系统实例 `recommender`；通过调用 `recommender.recommend` 方法，为指定用户生成推荐列表，返回值为 `RecommendedItem` 对象的列表，该列表包含推荐的资料项和相应的推荐评分；随后初始化一个空的 ArrayList（`cfItemIds`），用于存储推荐的资料 ID；遍历 `RecommendedItem` 列表，将每个推荐项的资料 ID 提取并添加到 `cfItemIds` 列表中；所有推荐项处理完毕后，返回包含推荐资料 ID 的列表，具体实现如图39所示。

图39 解析数据模型代码

最后，在 `topRecommendOfUserId` 方法中，根据用户 ID 为用户获取最终的推荐资料列表，完成个性化推荐功能的落地。方法执行流程如下：首先通过 `HttpServletRequest` 对象获取请求参数中的用户 ID（`userId`），并将其存储在 `uId` 字符串中；接下来初始化一个 `userId` 变量为 0，用于存储解析后的用户 ID；若 `uId` 的值不是字符串 "null"，则将其转换为 Long 类型并赋值给 `userId`，确保用户 ID 的有效性；若 `userId` 不为 0，则继续执行后续逻辑，处理有效用户的推荐请求。

为获取所有用户的资料收藏信息，调用 `favoriteService.list(new QueryWrapper<Favorite>().eq("target_type", 3))` 方法，得到用户收藏考研资料的列表 `List<Favorite>`；随后创建 `ArrayList<Recommend>`（`allRecommend`），用于收集所有用户的推荐信息，同时初始化一个空列表 `cfItemIds`，用于存储后续推荐的资料 ID；在循环中遍历收藏列表 list，为每个 `Favorite` 对象创建一个新的 `Recommend` 实例，设置对应的用户 ID、资料 ID（`target_id`）和推荐分数（设为 1，代表用户对该资料的偏好），并将这些推荐项添加到 `allRecommend` 列表中。

之后调用 `getDataModel(allRecommend)` 方法，生成数据模型 `dataModel`，用于后续的推荐计算；基于该数据模型，调用 `baseItem` 方法，传入用户 ID 和数据模型，获取推荐资料的 ID 列表 `cfItemIds`；若该列表不为 null，则创建一个新的 `ArrayList<Material>`，用于存储推荐的资料项；通过遍历 `cfItemIds` 列表，调用 `materialService.getById(l.longValue())` 从数据库中获取对应的 `Material` 对象，并将其添加到推荐资料列表中；最后返回包含推荐资料的列表，为用户展示个性化推荐内容，具体实现如图40所示。
